# D系統: フロー文脈レイヤー（fx-signal-monitor 拡張）

USD/JPY を対象に、「価格 ＝ フロー駆動の潜在フェアバリュー(FV) ＋ ノイズ」という状態空間モデルをカルマンフィルタで推定し、**フローで説明できない価格変化**を文脈情報として週次で可視化・通知するレイヤーです。

設計原則は「**文脈表示 → 検証 → シグナル化**」の三段階。本レイヤーは第1段階であり、D-4 の検証を通過するまで売買シグナルには昇格させません。

## モデル

```
状態方程式:  F_t = F_{t-1} + β·x_t + η_t,   η ~ N(0, Q)
観測方程式:  p_t = F_t + ε_t,               ε ~ N(0, R)

p_t : log(USDJPY)。COT報告基準日（火曜）に日次終値を merge_asof で整列
x_t : z156(−ΔCOT投機筋ネット) ＝ 円売りフロー強度（＋が円売り、β>0 が期待符号）
推定: (Q, R, β) を最尤法（Nelder-Mead 多点初期値）
```

出力はすべて**フィルタ値**（時点 t の値は t 以前の情報のみで計算）。スムーザーは表示に使いません。過去の線がきれいに見える代償に先読みが混入するためです。

### なぜ「gap」ではなく「3ヶ月累積イノベーション」が主指標か

1次元ローカルレベル型では、フィルタ後ギャップ `p_t − F_t|t` は `(1−K_t)·イノベーション` に恒等的に一致します。つまり gap は**毎週リセットされる量**で、「持続的な乖離」を測れません。そこで主指標を

- `innov_z` … 今週の未説明変動（サプライズ検出）
- `cum13` / `cum13_z` … **13週(≈3ヶ月)累積イノベーション** ＝ 過去3ヶ月の累積フロー関数で説明できない価格変化の総和

としています。`cum13` が当初構想「3ヶ月の累積流動性関数と現在価格の関係」の統計的に正しい実装形です。

### 状態判定

| 条件 | 状態/アラート |
|------|------|
| \|cum13_z\| < 1 | フロー整合圏 |
| 1 ≤ \|cum13_z\| < 2 | 乖離注意 |
| \|cum13_z\| ≥ 2 | 乖離大（方向付きアラート） |
| \|innov_z\| ≥ 2.5 | 今週サプライズ（イベント/介入等の確認促し） |
| net_z52 ≤ −2 | 投機筋の円ショート過熱（巻き戻しリスクの文脈） |
| net_z52 ≥ +2 | 投機筋の円ロング過熱 |

符号規約: COTネット（円先物）は **＋＝円ロング**。したがって net が深いマイナス＝キャリー方向の積み上がり。パネルの色は本邦チャート慣行に合わせ **赤＝円安側／青＝円高側**。

## セットアップ（fx-signal-monitor リポジトリへ）

1. 本ディレクトリの中身をリポジトリ直下にマージ（`docs/` が Pages ルートでない場合は
   `FLOW_OUTPUT_JSON` 環境変数と `flow_panel.html` の設置場所を合わせて変更）。
2. Secrets を確認: `DISCORD_WEBHOOK_URL`（既存流用可）、任意で `FRED_API_KEY`・`CFTC_APP_TOKEN`。
3. **初回は Actions の workflow_dispatch で手動実行**し、ログで COT/価格の取得成功と
   `docs/data/flow_context.json` のコミットを確認（サンドボックス外の実API疎通はここが初回になります）。
4. 動作確認は `FORCE_NOTIFY=1` を一時的に付けると通知経路までテストできます。
5. パネル単体のローカル確認: `python tests/make_demo_json.py` → `cd docs && python -m http.server 8000`
   → `http://localhost:8000/flow_panel.html`（file:// 直開きでは fetch が動きません）。
6. 既存ページに埋め込む場合は `flow_panel.html` の `<div class="fp">` 部と script を移植し、
   `--fp-*` CSS変数でテーマを合わせる。

## 出力JSON仕様（docs/data/flow_context.json）

- `model` … β, Q, R, logL, n（毎回全期間で再推定。`data/kf_params_history.csv` にドリフト履歴を蓄積）
- `latest` … 直近COT報告週の price_weekly / fair_value / cum13_pct / cum13_z / innov_z /
  net_spec_contracts / net_z52 / state
- `now` … 日次イマココ価格と対FV乖離%、`cot_staleness_days`（COT公表からの経過日数）。
  FVは報告週から**据え置き予測**のため参考値扱い。
- `series` … パネル用の週次系列（既定260週）
- `alerts` … 状態判定の文字列リスト（Discord通知は前回JSONとの差分がある時のみ）

## 制約と注意（重要）

- **COTは火曜集計・金曜公表（3日ラグ）**。祝日週は公表がずれることがある。今週の例で言えば、7/21〜22の163円台への急騰が反映されるのは 7/21 集計分＝**7/24(金)公表**、ジョブ反映は 7/25(土) 朝。
- 週次×COTという粒度上、効き得るのは**日次〜週次のスイング文脈**。1時間足のタイミングには使わない。
- 介入・地政学イベントは `innov_z` のスパイクとして「検出」はできるが「予測」はできない。
- 価格ソースはフォールバック構成だが、各社仕様変更があり得るため初回と月1回程度はログ確認を推奨。
- パラメータは毎回全期間MLEで再推定（D-4のバックテスト時は拡張窓での逐次再推定に切替が必要）。
- 多通貨展開時は `fetch_cot.py` の `CURRENCY_CODES_TODO` を実データで要確認。
